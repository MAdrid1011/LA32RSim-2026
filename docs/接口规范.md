# CPU 顶层接口规范

## 概述

LA32RSim-2026 仿真框架对被测 CPU 的顶层接口有明确约定。框架通过这些接口：

1. 提供时钟和复位信号
2. 实现内存读写（AXI 或简单总线）
3. 接收提交信息（用于 difftest 和 TRAP 检测）
4. 读取架构寄存器堆快照（用于 difftest 全量比较）

**核心设计原则：CPU 自己负责将物理寄存器映射回架构寄存器后输出，框架无需维护任何 rename 表。**
这意味着本框架支持任意发射宽度、任意微架构（顺序/乱序/超标量）的处理器。

## 1. Verilog 接口定义

### 1.1 完整接口（AXI 模式，NCOMMIT=2）

```verilog
module CPU (
    // ── 时钟与复位 ────────────────────────────────────────────────────────────
    input  wire        clock,           // 时钟，上升沿有效
    input  wire        reset,           // 同步复位，高电平有效

    // ── AXI 内存接口 ──────────────────────────────────────────────────────────
    // 读地址通道
    output wire        io_axi_ar_valid,
    input  wire        io_axi_ar_ready,
    output wire [31:0] io_axi_ar_addr,
    output wire [ 7:0] io_axi_ar_len,   // 突发长度 - 1
    output wire [ 2:0] io_axi_ar_size,  // 2^size = 字节数
    output wire [ 1:0] io_axi_ar_burst, // 00=FIXED, 01=INCR
    output wire [ 3:0] io_axi_ar_id,
    // 读数据通道
    input  wire        io_axi_r_valid,
    output wire        io_axi_r_ready,
    input  wire [31:0] io_axi_r_data,
    input  wire        io_axi_r_last,
    input  wire [ 1:0] io_axi_r_resp,
    input  wire [ 3:0] io_axi_r_id,
    // 写地址通道
    output wire        io_axi_aw_valid,
    input  wire        io_axi_aw_ready,
    output wire [31:0] io_axi_aw_addr,
    output wire [ 7:0] io_axi_aw_len,
    output wire [ 2:0] io_axi_aw_size,
    output wire [ 1:0] io_axi_aw_burst,
    output wire [ 3:0] io_axi_aw_id,
    // 写数据通道
    output wire        io_axi_w_valid,
    input  wire        io_axi_w_ready,
    output wire [31:0] io_axi_w_data,
    output wire [ 3:0] io_axi_w_strb,  // 字节使能
    output wire        io_axi_w_last,
    // 写响应通道
    input  wire        io_axi_b_valid,
    output wire        io_axi_b_ready,
    input  wire [ 1:0] io_axi_b_resp,
    input  wire [ 3:0] io_axi_b_id,

    // ── 提交接口（NCOMMIT=2，每个提交槽位独立） ───────────────────────────────
    // 第 0 个提交槽位
    output wire        io_cmt_0_valid,          // 本周期该槽位有指令提交
    output wire [31:0] io_cmt_0_pc,             // 提交指令的 PC
    output wire [31:0] io_cmt_0_inst,           // 提交指令编码（仅用于调试/追踪）
    output wire        io_cmt_0_rd_valid,        // 是否写目的寄存器
    output wire [ 4:0] io_cmt_0_rd,             // 目的架构寄存器编号（0-31）
    output wire        io_cmt_0_exception,       // 该提交是否携带异常
    output wire [ 5:0] io_cmt_0_exception_code, // 异常类型（LA32R 例外码）
    // 第 1 个提交槽位（NCOMMIT>=2 时存在）
    output wire        io_cmt_1_valid,
    output wire [31:0] io_cmt_1_pc,
    output wire [31:0] io_cmt_1_inst,
    output wire        io_cmt_1_rd_valid,
    output wire [ 4:0] io_cmt_1_rd,
    output wire        io_cmt_1_exception,
    output wire [ 5:0] io_cmt_1_exception_code,

    // ── 架构寄存器堆快照（difftest 用，每周期持续输出当前架构状态） ─────────────
    // CPU 负责维护物理→架构的映射，直接输出32个架构寄存器值
    output wire [31:0] io_cmt_rf_0,   // r0  (zero，始终为0)
    output wire [31:0] io_cmt_rf_1,   // r1  (ra)
    output wire [31:0] io_cmt_rf_2,   // r2  (tp)
    output wire [31:0] io_cmt_rf_3,   // r3  (sp)
    output wire [31:0] io_cmt_rf_4,   // r4  (a0)
    output wire [31:0] io_cmt_rf_5,   // r5  (a1)
    output wire [31:0] io_cmt_rf_6,   // r6  (a2)
    output wire [31:0] io_cmt_rf_7,   // r7  (a3)
    output wire [31:0] io_cmt_rf_8,   // r8  (a4)
    output wire [31:0] io_cmt_rf_9,   // r9  (a5)
    output wire [31:0] io_cmt_rf_10,  // r10 (a6)
    output wire [31:0] io_cmt_rf_11,  // r11 (a7)
    output wire [31:0] io_cmt_rf_12,  // r12 (t0)
    output wire [31:0] io_cmt_rf_13,  // r13 (t1)
    output wire [31:0] io_cmt_rf_14,  // r14 (t2)
    output wire [31:0] io_cmt_rf_15,  // r15 (t3)
    output wire [31:0] io_cmt_rf_16,  // r16 (t4)
    output wire [31:0] io_cmt_rf_17,  // r17 (t5)
    output wire [31:0] io_cmt_rf_18,  // r18 (t6)
    output wire [31:0] io_cmt_rf_19,  // r19 (t7)
    output wire [31:0] io_cmt_rf_20,  // r20 (t8)
    output wire [31:0] io_cmt_rf_21,  // r21 (reserved)
    output wire [31:0] io_cmt_rf_22,  // r22 (fp)
    output wire [31:0] io_cmt_rf_23,  // r23 (s0)
    output wire [31:0] io_cmt_rf_24,  // r24 (s1)
    output wire [31:0] io_cmt_rf_25,  // r25 (s2)
    output wire [31:0] io_cmt_rf_26,  // r26 (s3)
    output wire [31:0] io_cmt_rf_27,  // r27 (s4)
    output wire [31:0] io_cmt_rf_28,  // r28 (s5)
    output wire [31:0] io_cmt_rf_29,  // r29 (s6)
    output wire [31:0] io_cmt_rf_30,  // r30 (s7)
    output wire [31:0] io_cmt_rf_31   // r31 (s8)
);
```

### 1.2 简单总线接口（替换 AXI 部分）

当 `CONFIG_MEM_SIMPLE=y` 时，内存接口替换为：

```verilog
    // ── 简单单周期总线接口 ────────────────────────────────────────────────────
    output wire        io_mem_valid,    // CPU 发起访问请求
    input  wire        io_mem_ready,    // 仿真器表示本周期可以响应（始终为1）
    output wire [31:0] io_mem_addr,     // 访问地址
    output wire [31:0] io_mem_wdata,    // 写数据（读操作时忽略）
    output wire [ 3:0] io_mem_wstrb,    // 字节使能；全0表示读操作
    input  wire [31:0] io_mem_rdata,    // 读数据（写操作时忽略）
```

**时序约定（简单总线）：**
- `valid=1, wstrb=0`：读请求，`rdata` 在同一周期有效
- `valid=1, wstrb≠0`：写请求，下降沿到下一上升沿完成
- `ready` 始终为 1（单周期响应），CPU 不需要等待 ready

## 2. Chisel 接口定义

```scala
import chisel3._
import chisel3.util._

// 配置参数（与 Kconfig 保持一致）
object SimConfig {
  val NCOMMIT = 2  // 与 CONFIG_NCOMMIT 一致
}

// 单个提交端口
class CommitPort extends Bundle {
  val valid          = Output(Bool())
  val pc             = Output(UInt(32.W))
  val inst           = Output(UInt(32.W))   // 仅用于调试/追踪
  val rd_valid       = Output(Bool())
  val rd             = Output(UInt(5.W))    // 目的架构寄存器编号
  val exception      = Output(Bool())
  val exception_code = Output(UInt(6.W))   // LA32R 例外码
}

// AXI 读地址通道（CPU 侧 Master）
class AXIReadAddr extends Bundle {
  val valid = Output(Bool())
  val ready = Input(Bool())
  val addr  = Output(UInt(32.W))
  val len   = Output(UInt(8.W))
  val size  = Output(UInt(3.W))
  val burst = Output(UInt(2.W))
  val id    = Output(UInt(4.W))
}

// AXI 读数据通道
class AXIReadData extends Bundle {
  val valid = Input(Bool())
  val ready = Output(Bool())
  val data  = Input(UInt(32.W))
  val last  = Input(Bool())
  val resp  = Input(UInt(2.W))
  val id    = Input(UInt(4.W))
}

// AXI 写地址通道
class AXIWriteAddr extends Bundle {
  val valid = Output(Bool())
  val ready = Input(Bool())
  val addr  = Output(UInt(32.W))
  val len   = Output(UInt(8.W))
  val size  = Output(UInt(3.W))
  val burst = Output(UInt(2.W))
  val id    = Output(UInt(4.W))
}

// AXI 写数据通道
class AXIWriteData extends Bundle {
  val valid = Output(Bool())
  val ready = Input(Bool())
  val data  = Output(UInt(32.W))
  val strb  = Output(UInt(4.W))
  val last  = Output(Bool())
}

// AXI 写响应通道
class AXIWriteResp extends Bundle {
  val valid = Input(Bool())
  val ready = Output(Bool())
  val resp  = Input(UInt(2.W))
  val id    = Input(UInt(4.W))
}

// AXI 总线接口（完整）
class AXIBundle extends Bundle {
  val ar = new AXIReadAddr
  val r  = new AXIReadData
  val aw = new AXIWriteAddr
  val w  = new AXIWriteData
  val b  = new AXIWriteResp
}

// 简单总线接口
class SimpleBusBundle extends Bundle {
  val valid = Output(Bool())
  val ready = Input(Bool())
  val addr  = Output(UInt(32.W))
  val wdata = Output(UInt(32.W))
  val wstrb = Output(UInt(4.W))    // 0 = 读
  val rdata = Input(UInt(32.W))
}

// 调试/提交接口（固定，AXI 和简单总线共用）
class LA32RSimDebugIO extends Bundle {
  // NCOMMIT 个提交端口
  val cmt = Output(Vec(SimConfig.NCOMMIT, new CommitPort))
  // 当前架构寄存器堆（CPU 自维护物理→架构映射后输出）
  val rf  = Output(Vec(32, UInt(32.W)))
}

// CPU 顶层 IO（AXI 版本）
class CPUIO_AXI extends Bundle {
  val axi  = new AXIBundle
  val dbg  = new LA32RSimDebugIO
}

// CPU 顶层 IO（简单总线版本）
class CPUIO_Simple extends Bundle {
  val mem  = new SimpleBusBundle
  val dbg  = new LA32RSimDebugIO
}

// 顶层 CPU 模块示例（以 AXI 版本为例）
class CPU extends Module {
  val io = IO(new CPUIO_AXI)
  // ... 内部实现
}
```

## 3. 信号时序说明

### 3.1 提交接口时序

```
         ┌───┐   ┌───┐   ┌───┐   ┌───┐
clock ───┘   └───┘   └───┘   └───┘   └───

io_cmt_0_valid  ___/‾‾‾‾‾‾‾‾‾\___
io_cmt_0_pc     ===X  0x1c00004 X===
io_cmt_0_inst   ===X  0x00041022X===   (add.w r1, r1, r1)
io_cmt_0_rd_v   ___/‾‾‾‾‾‾‾‾‾\___
io_cmt_0_rd     ===X     1      X===
io_cmt_rf_1     ===X   new_val  X===   ← 写入后的新值，与 valid 同周期有效
```

**关键约定：**
- `io_cmt_rf_*` 必须在 `io_cmt_N_valid=1` 的**同一周期**反映最新提交后的架构寄存器值
- 对于乱序 CPU：退休（retire/commit）时更新架构寄存器文件并输出
- 对于多发射：同一周期多个 valid 槽位提交时，`io_cmt_rf_*` 反映**所有这些提交都完成后**的状态

### 3.2 AXI 内存接口时序（单次读）

```
         ┌───┐   ┌───┐   ┌───┐   ┌───┐
clock ───┘   └───┘   └───┘   └───┘   └───

ar_valid  ___/‾‾‾‾‾‾‾\___
ar_ready  ___________/‾‾‾\___          ← 仿真器控制
ar_addr   ===X  0x1c000000  X===

r_valid   _________/‾‾‾‾‾\___          ← 仿真器拉高
r_ready   ‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾          ← CPU 始终接受
r_data    =========X  data  X===
r_last    _________/‾‾‾‾‾\___
```

## 4. 实现指南

### 4.1 顺序单发射处理器

对于简单的顺序单发射流水线：

```verilog
// io_cmt_rf_* 直接连接到写端口后的寄存器文件读出
assign io_cmt_0_valid   = wb_valid;
assign io_cmt_0_pc      = wb_pc;
assign io_cmt_0_inst    = wb_inst;
assign io_cmt_0_rd_valid = wb_rd_valid;
assign io_cmt_0_rd      = wb_rd;
assign io_cmt_0_exception = 1'b0;
assign io_cmt_0_exception_code = 6'b0;

// 寄存器文件读端口（更新后的值）
genvar i;
generate
    for (i = 0; i < 32; i = i + 1) begin
        assign io_cmt_rf[i] = regfile[i];
    end
endgenerate
```

### 4.2 乱序多发射处理器

对于乱序处理器，退休时：

```verilog
// 在 ROB 退休（commit）阶段：
// 1. 将物理寄存器值写回架构寄存器文件（Architectural Register File, ARF）
// 2. io_cmt_rf_* 连接到 ARF 的读端口

assign io_cmt_0_valid   = rob_commit_0_valid;
assign io_cmt_0_pc      = rob_commit_0_pc;
// ...

// ARF 在 commit 阶段更新，io_cmt_rf_* 持续输出 ARF 当前值
for (genvar i = 0; i < 32; i++) begin
    assign io_cmt_rf[i] = arf[i];  // ARF = Architectural Register File
end
```

### 4.3 GOOD/BAD TRAP 约定

框架通过检测魔法字 `.word 0x80000000` 提交来判断程序结束：

- 检测到编码 `0x80000000` 的指令提交时终止仿真
- 检查 `io_cmt_rf_4`（r4/a0）的值：
  - `a0 == 0`：GOOD TRAP（测试通过）
  - `a0 != 0`：BAD TRAP（测试失败，返回值为错误码）

测试程序应在结束时执行：

```asm
    move    $r4, <exit_code>   # 0=通过，非0=失败
    .word   0x80000000         # 魔法字，通知仿真器程序结束
```

> **注意**：`0x80000000` 不是 LA32R 标准指令（LA32R 指令编码保证最高位为 0），
> 它是本框架专用的仿真退出魔法字，与 `break` 指令（`0x002a0000`）无关。
> base-port 运行时库中的 `halt()` 函数已封装此约定：
> ```c
> void halt(int code) {
>     asm volatile("move $r4, %0; .word 0x80000000" : : "r"(code));
> }
> ```
